# 開發交接流程（handoff）

這是本專案開發工作的固定收尾流程。整套日誌系統的核心紀律是：**對著 repo 的真實狀態作業，不憑印象、不杜撰。**

使用者這次工作的摘要（可能為空）：$ARGUMENTS

## 第一步：先抓取 repo 的真實狀態

在動筆寫任何日誌之前，先用 Bash 工具執行以下三道指令，取得真實狀態後再進行後續步驟：

- `git status`：確認目前哪些檔案有變動
- `git diff HEAD`：看清楚自上次 commit 以來的所有變動內容
- `git log -5 --oneline`：確認最近的提交紀錄

**規則：只根據上面的真實變動撰寫日誌。** 若不確定某項變動的意圖或結果，標記為「待確認」，不要自行補完或杜撰引擎沒有的能力。

## 第二步：更新 DEVLOG.md

把這次工作寫成一筆，**最新一筆放在檔案最上面**（最近的進度不必捲到底）。格式：

```
## YYYY-MM-DD — 一句話標題
狀態：這次結束後專案處於什麼狀態
做了什麼：
- …
技術決策：
- <決定> ← 理由 / 推翻了哪個舊決定
下一步：
- …
```

## 第三步：更新 FIXLOG.md（若這次有修 bug）

只有這次真的修了 bug 才寫。每一筆**必須包含全部五欄**，缺一不可：

```
## <bug 簡稱> (YYYY-MM-DD)
- 症狀：實際觀察到的錯誤行為
- 根因：真正的原因
- 錯誤直覺：當時看似正確、實際上更糟或無效的修法（← 這一欄是 FIXLOG 的靈魂，絕對不可省略）
- 正確修法：最後採用的解法
- 波及檔案：…
```

「錯誤直覺」這一欄是為了防止未來任何實例（包括聊天視窗的 Claude）重新建議那個看似對、其實更糟的修法。沒有這一欄的 FIXLOG 紀錄沒有價值。

## 第四步：更新 CLAUDE.md 的「引擎能力契約」（僅在引擎能力有變時）

只有當這次的工作**改變了引擎真正支援的能力邊界**（新增/移除欄位、機制、規則）時，才更新 CLAUDE.md 第 5 節「引擎能力契約」。

- 寫成「當前真相」——可直接遵守的規則，不要寫成「後來我們改成…」這種歷史敘述。
- 沒有改變引擎能力時，不要動這一節。

## 通用規則（每一步都適用）

1. **全程繁體中文。嚴禁簡體中文。**
2. **寫「當前真相」，不要讓讀者從歷史去推現狀。** 例：不要寫「後來改了掉落邏輯」，要寫「掉落邏輯目前是 push-append，不可改回覆寫」。
3. **只記錄這次真的發生的事**，不杜撰、不補完；不確定就標「待確認」。
4. **先把所有要寫入的草稿顯示給使用者，等使用者核可後才實際寫檔。** 不要未經確認就改檔。

## 第五步：產出要交給 Claude（聊天視窗）的提示詞

寫檔完成後，做最後一件事：把 CLAUDE.md 第 5 節「引擎能力契約」的**全文**讀出來，連同下面這段框架提示詞，一起輸出在終端機給使用者複製。這是要讓使用者貼給聊天視窗 Claude（負責設計劇本）用的，目的是讓劇本設計被限制在引擎真實能力的邊界內。

輸出格式如下（把方括號處替換成契約全文）：

```
這是目前的引擎能力契約（當前真相）。請依此設計劇本，不要使用契約中未列出的機制；若劇本需要新機制，請標記為「需引擎新增 <機制名>」交回給 Claude Code，不要自行用其他寫法硬湊。

[此處貼上 CLAUDE.md 第 5 節『引擎能力契約』全文]
```

輸出完這段後，本次交接流程結束。
